# Sitewide UI and WebGL Development Showcase Design

## Summary

TTI Intelligence のサイトを、次の二つの目的を同程度に達成する構成へ更新する。

1. 新規メンバーが活動内容を理解し、参加・問い合わせへ進みやすくする。
2. サークルの技術力と制作物を、サイト自身の表現力を通して強く印象づける。

初回スプラッシュは現在の見た目と最低表示時間を維持する。`/development` は短縮せず、React Three Fiber、Three.js、GSAP、ポストプロセスを使う最大演出版へ発展させる。それ以外の公開ページは、ナビゲーション、視線誘導、モバイル密度、操作性、アクセシビリティ、一貫性を改善する。

## Goals

- 入会・問い合わせと、技術・作品紹介をトップページ上で同格に扱う。
- 埋もれている Development、Game Community、Weekly Math を主要導線から発見できるようにする。
- `/development` を、サイトそのものが技術デモとして成立するスクロール連動3D体験にする。
- PC、スマートフォン、タブレットで、情報の優先順位と操作対象を明確にする。
- モバイルの長すぎるフッター、掲示板メタ情報、CLI練習の往復操作を改善する。
- カード、リンク、ボタン、読み込み・空・エラー状態の視覚・操作規則を統一する。
- キーボード、スクリーンリーダー、低モーション設定、低性能端末でも主要情報と操作を失わない。

## Non-goals

- 初回スプラッシュの削除、短縮、ブランドビジュアルの再設計は行わない。
- Development の通常ページ化や、長いスクロール体験の撤廃は行わない。
- Firebase のデータモデル、認証方式、投稿内容、管理機能の業務フローは変更しない。
- サイト全体を別のデザインシステムやUIライブラリへ置き換えない。
- Development 以外へ WebGL を常駐させない。

## Implementation decomposition

この仕様はサイト全体の完成像を定義する。実装とレビューは、互いに検証可能な次の3サブプロジェクトへ分ける。

1. **Site shell and UI foundation**
   - スプラッシュの inert 化
   - ナビゲーションとFooter
   - ButtonLink、InteractiveCard、Card余白
   - 色・文字・フォーカス・フォーム・Dialog・状態表示
   - 後続サブプロジェクトが利用する共通インターフェースを確定する
2. **Public page responsive UX**
   - Home、About、Game Community、Weekly Math、News、Appsと各ツール、Board、Contact、Adminの対象改善
   - Site shell and UI foundation の共通部品を利用する
3. **Development WebGL showcase**
   - 3D依存、品質判定、7章のCanvasシーン、進捗UI、静的フォールバック
   - Site shell and UI foundation のナビゲーション、ボタン、フォーカス規則を利用する

Site shell and UI foundation を最初に完了する。Public page responsive UX と Development WebGL showcase は、その共通インターフェースが確定した後、別々の実装計画とテストゲートで進められる。

## Experience principles

### Balanced conversion

トップページの最初の主要導線は「サークルについて」と「開発を見る」の二つとする。参加・問い合わせの最終CTAは従来どおりページ下部にも残し、技術紹介と入会導線のどちらか一方だけを優先しない。

### Continuous visual feedback

長いスクロール演出では、どのスクロール位置でも次のいずれかを表示する。

- 主役となる3Dオブジェクト
- 背景の粒子、ノード、コード断片、データライン
- 章名、進捗、次章の予告
- 操作や移動を示す短いガイド

画面全体が無情報に見える区間は作らない。

### Progressive enhancement

WebGLが利用できない場合、低性能判定の場合、または `prefers-reduced-motion: reduce` の場合も、同じ章タイトル、説明、技術スタック、開発フロー、CTAを静的HTMLで提供する。3D表現は情報の唯一の担い手にしない。

## Information architecture

### Primary navigation

PCヘッダーを次の構成へ変更する。

- Home
- About Us
- Activities
- News
- Apps
- Board
- Contact

Activities はドロップダウンとして、次のリンクを持つ。

- Development
- Game Community
- Weekly Math

Activities 配下のルートでは親項目もアクティブ表示にする。モバイルメニューではドロップダウン操作を要求せず、「活動」見出しの下へ3リンクを常時表示する。

### Footer

PCではブランド・連絡先、公開ページメニュー、活動メニューの3領域を表示する。管理者ページは主要メニューから外し、著作権行付近の控えめな補助リンクへ移す。

スマートフォンではブランド説明とSNSを常時表示し、「連絡先」「メニュー」をネイティブな開閉状態を持つアコーディオンにする。Contact ページでは本文に表示済みのメール情報をフッター内で簡略化する。ツール系ページでも同じコンパクトフッターを使用する。

## Development maximum-production experience

### Technology

- `three`
- `@react-three/fiber`
- `@react-three/drei`
- `@react-three/postprocessing`
- 既存の `gsap`

Development の遅延ロード境界内だけでこれらを読み込み、他のルートの初期バンドルへ含めない。

### Scene architecture

既存の7章構成を維持し、単一の固定Canvasと、章ごとのHTMLオーバーレイを連動させる。

1. **AI Core Formation** — 点群とコード片が集まり、発光するAIコアを形成する。
2. **Technology Constellation** — 技術スタックが3D空間のノードとして展開し、関連線で接続される。
3. **MCP Data Corridor** — ツール、モデル、データ間をパケットが移動し、MCPの接続概念を見せる。
4. **Build Chamber** — UI、コード、データ層が空間上で組み上がり、Webページのプレビューになる。
5. **Tool Orbit** — 日常的に利用するAI・開発ツールがコアの周囲を軌道運動する。
6. **Workflow Pipeline** — 構想、実装、確認、公開を、発光する経路と状態遷移で示す。
7. **Reconstruction Finale** — これまでの要素が現在のページへ再構成され、「このページも作品」というメッセージとCTAへ着地する。

### Camera and scroll behavior

- ページ全体の長いスクロール体験は維持する。
- 各章は連続したカメラパスでつなぎ、章境界で完全な暗転や空白を作らない。
- 画面端に章番号と全体進捗を常時表示する。
- 進捗UIから各章へ移動できるが、通常スクロールを妨げない。
- 冒頭に「スクロールして探索」、進捗UIに「最終章へ」、最終章に「Apps」「Contact」「Discord」のCTAを置く。
- 第4章にも中間CTAとして「制作物を見る」を置き、最終章まで待たなくても回遊できるようにする。

### Visual treatment

- Bloom、発光、軽いノイズ、走査線、控えめな色収差を利用する。
- 色収差とノイズはテキストには適用しない。
- 背景はネイビーから紫への既存トーンを維持する。
- マウスとタッチ位置に応じて、カメラまたは粒子へ小さな視差を与える。
- フォーカス中のリンク・ボタンは3Dエフェクトより前面に表示する。
- 重要な説明文は最小16px、補助文は最小13pxとする。

### Performance and fallback

- PC高性能: DPR上限2、ポストプロセス全機能、最大粒子数。
- PC低性能・一般スマホ: DPR上限1.25、粒子数削減、色収差と高コストブラーを無効化。
- 低性能スマホ・Save-Data・Reduced Motion: Canvasを描画せず、静的な章カードと軽量CSS背景を表示。
- Canvasはページ非表示時とタブ非アクティブ時に停止する。
- WebGL初期化失敗時はエラー表示を出さず、静的版へ自動切り替えする。
- Development の3Dコードは独立チャンクとし、他ページのLCPに影響させない。

## Shared UI changes

### Buttons and links

- `Button` はボタン操作専用とする。
- ページ遷移には、同じ見た目を持つ `ButtonLink` または `asChild` 相当の単一インタラクティブ要素を使う。
- リンク内にネイティブ `button` を入れない。
- タッチ操作対象は原則44×44px以上とする。
- フォーカスリングは共通レシピへ統一する。

### Interactive cards

`InteractiveCard` を共通化し、次を保証する。

- カード全体がリンクの場合は、カード全体を一つのリンクにする。
- 操作できないカードには、ホバー浮上や矢印を表示しない。
- PCではホバーとフォーカス、スマホでは常時表示の方向キューを用意する。
- タイトル、状態、要約、明示的なアクションの順に読む視線順序を統一する。

### Card spacing

`Card` と `CardContent` のどちらか一方だけがパディングを所有する。構造化カードは `Card padding="none"` とし、`CardContent` にレスポンシブな余白を設定する。320px幅でも本文領域を過度に狭めない。

### Color and typography

- 通常サイズの補助テキストはWCAG AAの4.5:1以上を目標にする。
- 現在の `#86868B` と30%白は装飾専用とし、読ませる本文・日付には濃いセカンダリートークンを使う。
- ダークテーマのページ、カード、インセット、没入型背景に意味ベースのサーフェストークンを定義する。
- 見出しは `clamp()` を用いた滑らかなスケールにし、1024pxでの急激な倍増を避ける。

### Loading, empty, error, feedback

- `LoadingState`、`EmptyState`、`ErrorState` の共通パターンを用意する。
- 一覧の読み込みでは内容に近いスケルトンを使う。
- 空状態には理由と次の操作を表示する。
- 再取得可能なエラーには「再試行」を表示する。
- 成功・一般エラーは既存Toastへ統一し、入力エラーはフィールド直下へ表示する。
- 破壊操作はブラウザ `confirm()` ではなく、アプリ内確認ダイアログを使用する。

### Navigation and accessibility

- ページ先頭に「本文へスキップ」を追加する。
- SPA遷移時は新しい `<main>` または主要見出しへフォーカスを移し、ページ名を通知する。
- モバイルメニューは Escape、初期フォーカス、フォーカストラップ、背景の inert、元のボタンへのフォーカス復帰を持つ。
- Dialog はタイトルと説明を `aria-labelledby` / `aria-describedby` で関連付ける。
- Input と Textarea はラベル、`aria-invalid`、エラー説明の関連付けを必須化する。

## Page-specific changes

### Home

- ヒーローの主要CTAを「サークルについて」と「開発を見る」にする。
- 下部の入会問い合わせCTAは維持する。
- スマホでも主要ボタンを44px以上にする。
- 動画カルーセルの左右操作とカードの視覚キューをタッチ向けに調整する。
- 初回スプラッシュの見た目と最低2.2秒の表示は維持し、表示中は背後のアプリを inert にする。Reduced Motionでは静止画として同じ時間を表示する。

### About

- ヒーロー直後に4活動へのジャンプナビを置く。
- 固定高と絶対配置をやめ、スマホではコピーと操作を通常フローにする。
- 次回イベントへの導線を活動カード群の前後で一度ずつ表示する。
- FAQへ `aria-expanded` と関連する回答IDを追加する。

### Game Community

- ヒーローへ「Discordに参加」と「活動について問い合わせる」を追加する。
- Featured Games は実リンクがないため、ホバー浮上と矢印を外して情報カードとして扱う。
- 下部CTAは繰り返し導線として維持する。

### Weekly Math

- 一覧カードの本文を短い要約へ制限し、「問題を開く」を常時表示する。
- ステータス色だけに依存せず、ラベルでも公開状態を伝える。
- 数式が幅を超える場合は数式領域だけ横スクロール可能にする。

### News

- 記事が存在するカテゴリだけを表示する。
- スマホではタグを検索・カテゴリの直後へ移動する。
- 検索入力へ視覚的またはスクリーンリーダー用ラベルを付ける。
- 固定記事と最新記事の並びは、固定ラベルを保ちながら日付順が理解できる見せ方にする。

### Apps

- PCは4カードを2×2で均等に配置する。
- 公開中、調整中の状態と利用可否を一致させる。
- 内部リンクに外部リンクアイコンを使わない。
- アプリ詳細ページへ共通の `AppToolHeader` を導入し、戻る、英字アイブロウ、タイトル、説明、主要操作の順序を統一する。

### Table Tennis

- スマホでタイトル、PDF/印刷操作、人数・クール数を縦方向に整理する。
- 数値入力は16px以上にしてiOSの自動ズームを避ける。
- 「PDF出力」は実際の挙動に合わせて「印刷・PDF保存」へ変更する。
- ラウンド選択へ選択状態のセマンティクスを追加する。

### Color Sort

- 色だけでなく模様または記号でもレイヤーを識別できるようにする。
- ボトルの内容、選択状態、有効な移動先を読み上げ可能にする。
- 長い操作ガイドとエラー理由を切り詰めず、折り返して表示する。
- 盤面上部の主要操作は44px以上を維持する。

### CLI Practice

- スマホは「教材」「実践」「ファイル」の固定セグメントで主領域を切り替え、説明とターミナルの長い往復をなくす。
- 現在のステップと進行数はセグメントの上に固定表示する。
- ターミナル、エディタ、検索入力を16px以上にする。
- ターミナルとエディタへ明確なフォーカスリングを追加する。
- PCの3カラム構成は維持する。

### Board

- スマホでは投稿者・日時と、コメント・いいねを別行にする。
- 長い投稿者名は `min-width: 0` と折り返しまたは省略で吸収する。
- 管理者操作はカードリンクの外へ出し、スマホでは折り返す。
- Likeへ `aria-pressed` とラベルを追加する。
- 読み込み、空、エラー状態を共通状態コンポーネントへ合わせる。

### Contact

- フォーム直前に「通常の返信目安」「入力情報の利用目的」「必須項目」を短く表示する。
- スマホの送信ボタンはフォーム末尾で明確にし、44px以上を維持する。
- フッター内の重複連絡情報は簡略表示にする。

### Admin

- 共有UI変更に伴うコントラスト、ボタン、フォーム、ダイアログ、フィードバックを適用する。
- 管理業務の情報設計や機能追加は行わない。
- スマホでメールアドレスとログアウト操作が衝突しないよう縦積みにする。

## Data flow and state

- ナビゲーション設定は `siteConfig` で、主要リンクと Activities 配下を別配列として管理する。
- Development の章進捗は既存のスクロール進捗を正規化し、HTMLオーバーレイと3Dシーンが同じ値を読む。
- 3D品質設定は端末特性、Save-Data、Reduced Motion、WebGL初期化結果から一度決定し、描画中に不安定に切り替えない。
- CLI Practice のモバイルセグメントは表示領域だけを切り替え、既存のチュートリアル・仮想ファイル・ターミナル状態を保持する。
- Footer の開閉状態はページ遷移後に閉じた初期状態へ戻す。

## Error handling

- Development の3Dチャンク取得またはWebGL初期化に失敗した場合、静的版を即時表示する。
- Firebase一覧取得失敗時は、共通ErrorStateと再試行操作を表示する。
- Toastは一時的な結果に限定し、修正が必要なフォームエラーは入力欄の近くへ残す。
- 共有ダイアログは閉じた後に起点要素へフォーカスを戻す。

## Testing strategy

### Automated component tests

- Activities ナビゲーションの表示、親アクティブ状態、モバイル展開。
- コンパクトFooterの開閉、リンク構成、管理者リンクの位置。
- ButtonLink と InteractiveCard が入れ子の操作要素を生成しないこと。
- Boardのメタ情報がスマホ用構造へ分割されること。
- Newsの空カテゴリ非表示とタグ順序。
- CLI Practiceのモバイルセグメント切り替え時に状態が保持されること。
- Color Sortの読み上げラベル、選択状態、非色覚キュー。
- Developmentの品質判定、静的フォールバック、章進捗計算。

### Existing test suite

- Vitestの全テスト。
- TypeScriptビルド。
- ESLint。

### Browser verification

- 1440×1000、1024×768、390×844、320×700。
- ライト、ダーク、Reduced Motion。
- Developmentは高品質WebGL、低品質WebGL、静的フォールバックを確認する。
- キーボードだけでヘッダー、メニュー、カード、ダイアログ、フォーム、Developmentのスキップを操作する。
- 全公開ルートで横スクロールが発生しないことを確認する。
- Development以外の初期バンドルに3D依存が含まれないことをビルド出力で確認する。

## Acceptance criteria

- 初回スプラッシュのブランド表現と最低表示時間が維持される。
- Homeのファーストビューから参加情報とDevelopmentの両方へ進める。
- Development、Game Community、Weekly MathがPC・スマホの主要ナビから到達できる。
- Developmentの全スクロール区間で、主役、背景モーション、章情報のいずれかが見える。
- Developmentは7章、本格3D、ポストプロセス、スクロールカメラ、途中CTA、最終CTAを持つ。
- Reduced MotionまたはWebGL失敗時にも7章の情報と全CTAへ到達できる。
- 320px幅で横スクロールがなく、主要操作が44px以上である。
- スマホのBoard日時が不自然な縦分割にならない。
- スマホのCLI Practiceで教材、ターミナル、ファイルを長距離スクロールせず切り替えられる。
- 読ませる補助テキストがライト・ダーク双方で十分なコントラストを持つ。
- リンクとボタンの不正な入れ子が解消される。
- 既存テスト、追加テスト、ビルド、Lintが成功する。
