# TTI Intelligence サイトAIガイド 設計

## 目的

TTI Intelligence の公開サイトに、初めて訪れた人と既存メンバーの両方を対象とするAIガイドを追加する。利用者の質問に、サイト内の公開情報とリポジトリで管理する案内・FAQだけを根拠として短く答え、目的に合う実在ページへ案内する。

初期版は「何でも答えるAI」ではなく、サイト内ナビゲーションを助ける小さな案内役に限定する。OpenAI APIの利用残高が5ドル程度でも運用を開始できるよう、送信コンテキスト、出力長、呼び出し回数を制限する。

## 成功条件

- 利用者が活動内容、参加方法、各機能の場所などを自然文で質問できる。
- 回答に含まれるリンクは、コード側で定義した実在ページだけに限定される。
- 「お知らせ」から「今週の数学」などへ移動しても、同じタブ内の会話と開閉状態が維持される。
- AIガイドを邪魔に感じる利用者は、そのタブを再読み込みするまで右下ボタンを完全に非表示にできる。
- OpenAI APIキーはブラウザへ送られず、会話履歴はアプリのサーバーやデータベースに保存されない。
- 初期設定では1日100回を超えてOpenAI APIを呼ばず、残高の急減を防ぐ。

## 確定した利用体験

### 右下ボタン

- 公開ページの右下に、A1案の静止した丸ボタンを表示する。
- 大きさはPCで44px、スマートフォンで48pxとし、押せる領域とアクセシビリティを確保する。
- 自動表示、勝手な展開、点滅、バウンド、継続的なアニメーションは行わない。
- ボタンには見た目だけに依存しない `aria-label="AIガイドを開く"` を付ける。
- `/admin` 以下では表示しない。その他の公開ルートでは共通して表示する。
- ウィジェットは通常コンテンツより上、Header、DevHeader、モバイルメニュー、初回スプラッシュより下の重なり順に置く。既存の右下Toastとは併用せず、AIのエラーは会話パネル内に表示する。

### 会話パネル

- ボタンを押したときだけ開く。
- PCでは右下に約360px幅のパネル、スマートフォンでは画面下から開くボトムシートとして表示する。
- 初期画面には「活動内容を知りたい」「参加方法を知りたい」「目的のページを探す」の3つの候補を表示する。
- ヘッダーの閉じるボタンはパネルだけを閉じ、右下ボタンは残す。
- パネルヘッダーのオーバーフローメニュー（`…`）に「このタブで右下ボタンを非表示」を用意する。選択するとパネルと右下ボタンを即座に消す。
- 非表示後に「元に戻す」、メニュー項目、フッター項目などの代替入口は表示しない。
- サイト内のReact Router遷移では非表示状態を維持する。再読み込み、新しいタブ、またはサイトを開き直した場合は再表示する。

### ページ遷移と会話状態

- AIガイドの状態は `Layout` 配下の共通プロバイダーで管理し、個別ページの `Outlet` より長く生存させる。
- サイト内リンクはReact Routerで遷移するため、会話履歴、パネルの開閉、送信中状態、タブ内の非表示状態を維持する。
- ルートが変わるたびに `currentPath` を更新し、次の質問では移動後のページを現在地としてLambdaへ送る。
- ブラウザの再読み込みやタブ終了ではReactメモリが破棄され、会話履歴もリセットされる。
- `localStorage` と `sessionStorage` は会話、開閉、非表示の保存に使用しない。

## システム構成

```text
React UI
  ├─ AssistantProvider: 会話・開閉・タブ内非表示
  ├─ AssistantWidget: 右下ボタンとレスポンシブなパネル
  └─ assistantApi: POST /assistant
            │
            ▼
API Gateway
  ├─ エンドポイント単位のスロットリング
  └─ Lambda統合
            │
            ▼
Assistant Lambda
  ├─ 入力検証
  ├─ ローカル案内データから関連項目を最大5件選択
  ├─ 関連項目がなければ固定のContact案内を返す
  ├─ Secrets ManagerからAPIキーを取得
  ├─ 利用回数枠を原子的に確保
  ├─ OpenAI Responses APIを呼び出し
  └─ ページIDを検証済みURLへ変換
```

既存のReact 19、React Router、AWS CDK、API Gateway、Lambdaの構成を維持する。フロントエンドからOpenAIへ直接アクセスさせない。

## フロントエンドの責務

### コンポーネント境界

- `AssistantProvider`
  - メッセージ配列、開閉、送信中、タブ内非表示を管理する。
  - リロードをまたぐ保存は行わない。
- `AssistantWidget`
  - 丸ボタン、デスクトップパネル、モバイルボトムシートを表示する。
  - `/admin` 以下では描画しない。
- `AssistantConversation`
  - メッセージ、候補ボタン、ページリンク、入力欄、エラーを表示する。
- `assistantApi`
  - APIのリクエスト・レスポンス型、タイムアウト、エラー変換を一箇所へ集約する。

巨大な単一コンポーネントにはせず、状態管理、表示、通信を独立してテストできる単位に分ける。

### 入力と表示

- 質問は空白を除いて1〜500文字とする。
- 送信中は同じ入力の二重送信を無効にする。
- APIへ渡す履歴は直近6往復、最大12メッセージまでとする。
- AIの回答はプレーンテキストとして描画し、モデル生成HTMLは描画しない。
- ページリンクは回答本文とは別のリンクカードとして表示する。
- リンクカードは必ずReact Routerの `Link` または `useNavigate` を使い、通常の `<a href>` による再読み込みで会話を失わないようにする。
- `Escape` でパネルを閉じ、閉じた後は右下ボタンへフォーカスを戻す。
- 新しい回答は `aria-live` で通知し、キーボードだけでも全操作を完了できるようにする。
- デスクトップパネルは非モーダルな `role="dialog"` とし、開いたとき入力欄へフォーカスする。モバイルのボトムシートは `role="dialog"`、`aria-modal="true"`、背景の操作抑止、フォーカストラップ、本文スクロールロックを備える。

## API契約

### リクエスト

`POST /assistant`

```json
{
  "message": "今週の数学はどこ？",
  "currentPath": "/news",
  "sessionId": "ブラウザメモリ上で生成したUUID",
  "history": [
    { "role": "user", "content": "活動内容を知りたい" },
    { "role": "assistant", "content": "About Usで確認できます。" }
  ]
}
```

- `currentPath` には `useLocation().pathname` だけを送り、クエリとハッシュは含めない。Lambda側では静的ルートに加えて `/news/:slug`、`/weekly-math/:weekKey`、`/weekly-math/:weekKey/solution`、`/board/:id` などの既知テンプレートへ照合し、未知のパスは「現在ページ不明」として扱う。
- `sessionId` は連続送信制限の単位にだけ使用し、会話の復元や利用者追跡には使用しない。
- `sessionId` は利用者が再生成できるため、セッション上限は誤操作や通常ブラウザからの連打を抑える補助策と位置づける。予算の強制上限は日次全体カウンターで担保する。
- 履歴はロール、件数、各本文長、合計文字数をサーバー側でも再検証する。各本文は最大800文字、履歴全体は最大8,000文字とし、超過時はHTTP 400を返す。

### レスポンス

```json
{
  "answer": "今週の数学は、問題一覧から確認できます。",
  "links": [
    { "pageId": "weekly-math", "title": "今週の数学", "href": "/weekly-math" }
  ]
}
```

OpenAIからは構造化出力で `answer` と `pageIds` を受け取る。`answer` は1〜500文字、`pageIds` は重複なしで最大3件とする。Lambdaは未知のページIDに加え、その質問でOpenAIへ送っていないページIDも破棄する。許可集合は今回選択した最大5件と固定のContactページだけとし、登録されたタイトルとパスを使って `links` を組み立てる。モデルが自由なURLを返してもクライアントへ渡さない。

## 案内データと検索

### データ形式

案内データは `lambdas/public/assistant/knowledge/site-guide.json` で管理し、初期版では管理画面を作らない。各項目は次を持つ。

```json
{
  "id": "weekly-math",
  "route": "/weekly-math",
  "title": "今週の数学",
  "summary": "公開されている数学問題の一覧",
  "audiences": ["visitor", "member"],
  "keywords": ["数学", "問題", "今週", "解説"],
  "faqs": [
    { "question": "問題はどこ？", "answer": "今週の数学の一覧から選べます。" }
  ],
  "relatedPageIds": ["home", "contact"]
}
```

対象にはHome、About Us、News、Apps、Development、Board、Contact、Game Community、Weekly Mathに加え、Table Tennis Match Maker、Color Sort Puzzle、CLI Practiceの固定ページを含める。FAQは活動内容、参加方法、連絡方法、各ページの用途など、公開してよい固定情報に限定する。

案内データを変更するときは、該当公開ページの表示内容も同時に確認する。特にAbout Us内の既存FAQとルート一覧は重複しやすいため、登録ルートが実在することと主要FAQの回答が公開コピーと矛盾しないことをテストする。

### 関連項目の選択

- 質問はUnicode NFKC、英字小文字化、前後空白除去、連続空白統合で正規化する。
- タイトルの完全一致を8点、タイトルの部分一致を5点、キーワード一致を3点、FAQ質問の語句一致を2点、現在ページ一致を1点として加算する。
- 同点は案内データ内の安定した `id` 順で決め、最低スコア3点以上の上位最大5項目だけをOpenAIへ送る。現在ページの1点だけでは採用しない。
- 3点以上の項目がない場合はOpenAIを呼ばず、「案内できる情報が見つかりませんでした。お問い合わせページをご利用ください」という固定回答とContactリンクを返す。
- 初期版では埋め込み、Vector Store、File Search、外部Web検索を使用しない。
- News、Weekly Math、Boardの動的な個別記事本文は読み込まず、一覧ページまたは該当機能への案内に限定する。

## OpenAI Responses API

- モデルはユーザーが選択した `gpt-5.6-luna` を使用する。
- Responses APIを非ストリーミングで呼び出す。
- `store: false` とし、OpenAI上のResponseオブジェクトをアプリケーション状態として保存しない。
- 推論設定は `reasoning.effort: "none"`、JSONの構造分を含む出力上限は `max_output_tokens: 600` とする。
- Structured Outputsを使い、回答本文と候補ページIDのスキーマを固定する。
- Web Search、File Search、Code Interpreterなどのツールは有効にしない。
- 1回の利用者送信につきOpenAI呼び出しは1回とし、自動再試行は行わない。
- システム指示には、提供された案内データだけを根拠にすること、不明な場合は推測しないこと、短く答えることを明記する。
- `message`、`history`、`currentPath` はすべて信用できない利用者入力として扱い、会話内の指示でシステム範囲や案内データを上書きさせない。
- 回答根拠が見つからない場合は、分からない旨とContactページへの案内を返す。

2026年7月16日時点の公式情報では、GPT-5.6 LunaはResponses APIとStructured Outputsをサポートし、テキスト料金は入力100万トークンあたり1ドル、出力100万トークンあたり6ドルである。価格や提供条件は変更され得るため、デプロイ前にも公式モデルページを確認する。

- [GPT-5.6 Luna公式モデルページ](https://developers.openai.com/api/docs/models/gpt-5.6-luna)
- [Responses APIリファレンス](https://platform.openai.com/docs/api-reference/responses)

## Lambdaの責務

- `index.ts`: API Gatewayイベントの受付、各処理の呼び出し、HTTPレスポンス変換だけを担当する。
- `validation.ts`: リクエスト、履歴、UUID、パス、OpenAI構造化出力を検証する。
- `knowledge.ts`: JSON読込、正規化、スコアリング、ページIDと固定URLの対応を担当する。
- `quota.ts`: 日次・セッション窓カウンターのトランザクションと429判定を担当する。
- `openai.ts`: Secrets Manager、20秒Abort、Responses APIリクエスト、上流エラー分類を担当する。
- `types.ts`: API契約と内部データ型を定義する。

ハンドラーから検索、予算制御、OpenAI通信を分離し、それぞれを外部サービスなしで単体テストできるようにする。

## 秘密情報とプライバシー

- OpenAI APIキーはAWS Secrets Managerの `tti-ai/openai-api-key` に `{"apiKey":"..."}` の形式で事前登録し、Assistant Lambdaだけに読み取り権限を与える。
- Lambdaはシークレット値をコールドスタート後のプロセスメモリへキャッシュし、通常リクエストごとのSecrets Manager読み取りを避ける。
- APIキーをVite環境変数、配布JavaScript、リポジトリ、APIレスポンスへ含めない。
- Lambdaは会話本文をCloudWatchへ記録しない。記録するのはリクエストID、成功・失敗、処理時間、トークン使用量などの運用情報に限定する。
- DynamoDBへ保存するのは利用回数カウンターだけで、質問、回答、会話履歴は保存しない。
- フロントエンドの会話履歴はReactメモリだけに存在し、再読み込みまたはタブ終了で破棄される。

## 予算保護と連続送信制限

- 専用DynamoDBテーブルへ、JST日付単位の全体カウンターと短時間のセッションカウンターを保存する。
- 全体上限は環境変数で変更可能にし、初期値を1日100回とする。
- セッション上限は10分間に20回とし、通常の会話を妨げず連続呼び出しを抑える。
- 日次項目とセッション窓項目は `TransactWriteItems` の条件付き更新で同時に枠を確保し、並行リクエストでも上限を超えてOpenAIを呼ばない。
- 枠はOpenAI呼び出しの直前に消費する。上流エラーやタイムアウトでも返却せず、1リクエストが複数回分を消費する自動再試行は行わない。
- API GatewayのStage `methodOptions` で `/assistant/POST` だけを毎秒2リクエスト、バースト4に制限する。
- フロントエンドは送信中の入力を無効にし、意図しない二重送信を防ぐ。
- 上限に達した場合はHTTP 429を返し、「本日のAIガイド利用上限に達しました。通常のメニューをご利用ください」と表示する。
- カウンターにはTTLを設定し、会話内容や個人情報を保存しない。
- セッション窓の項目は1時間後、日次項目は48時間後をTTLとする。TTL削除が遅れても日付・窓を含む別キーを使うため、新しい上限判定には影響しない。

## エラー処理

- 不正JSON、空入力、500文字超過、過大な履歴、未知のロールはHTTP 400とする。
- 利用回数または連続送信の上限はHTTP 429とする。
- OpenAIの認証・上限・一時障害は内部ログで分類し、ブラウザには秘密情報を含まない共通メッセージを返す。
- OpenAIへの `fetch` は20秒でAbortし、捕捉してHTTP 504を返す。その他の上流障害はHTTP 502とする。
- 構造化出力が検証できない場合は自由形式をそのまま返さず、安全なフォールバック回答を返す。
- フロントエンドは失敗した質問を入力欄へ戻し、利用者が修正または再送できるようにする。
- API障害中も通常のサイトナビゲーションは影響を受けない。

## AWS/CDK変更

- Assistant Lambdaを `aws-lambda-nodejs.NodejsFunction` によるNode.js 22のバンドル済み関数として追加する。既存の未コンパイル `.ts` を `Code.fromAsset` で渡す方式は流用しない。
- OpenAI呼び出しにはNode.js 22の `fetch` を使い、新しいOpenAIランタイム依存を増やさない。AWS SDK v3はLambdaランタイム提供モジュールとして外部化する。
- `POST /assistant` を既存API Gatewayへ追加する。
- 既存API全体の `ALL_ORIGINS` をそのまま継承せず、AssistantリソースのOPTIONSと実レスポンスでは `https://tti-intel.com` と `http://localhost:5173` だけを許可する。CORS自体は不正呼び出し対策ではないため、予算保護は日次カウンターで担保する。
- OpenAIシークレット `tti-ai/openai-api-key` を名前で参照し、Lambdaへ `secretsmanager:GetSecretValue` の最小権限を付与する。
- 利用回数専用DynamoDBテーブルを従量課金、TTL有効で追加し、Lambdaへ必要な更新権限だけを付与する。
- Lambdaのタイムアウトを25秒、フロントエンドの通信タイムアウトを28秒とし、API Gatewayの上限内でLambdaの終了を待てるようにする。
- フロントエンドには公開可能なAPI GatewayのベースURLを `VITE_API_BASE_URL` として設定し、`assistantApi` が `/assistant` を付加する。OpenAIキーを `VITE_*` へ設定しない。
- CloudWatchではエラー率、429、処理時間、OpenAIトークン使用量を確認できるようにする。

## デプロイ手順

1. Secrets Managerへ `tti-ai/openai-api-key` を事前登録する。
2. CDKのbuildとsynthを通し、CDK deployでLambda、DynamoDB、API Gatewayを反映する。
3. CDK出力のAPI GatewayベースURLをAmplifyの `VITE_API_BASE_URL` に設定する。
4. フロントエンドを再ビルド・デプロイする。
5. 日次上限を低いまま、明示的なスモークテストを1回だけ行う。

現在のAmplify設定はフロントエンドだけをビルドするため、CDK/Lambdaのデプロイは別工程として扱う。

## テスト方針

### フロントエンド

- 丸ボタンが自動で開かず、クリック時だけパネルを開く。
- PCとモバイルの表示、キーボード操作、初期フォーカス、モバイルのフォーカストラップ・背景操作抑止・スクロールロック、閉じた後のフォーカス復帰を確認する。
- React Routerでページ遷移しても会話、開閉、非表示状態が維持される。
- 「このタブで非表示」にすると代替入口や元に戻す通知を描画しない。
- コンポーネントを再マウントすると右下ボタンと空の会話が復元される。
- ボタンまたはパネルを開いただけではAPIを呼ばず、利用者が質問を送ったときだけ呼び出す。
- 二重送信、500文字制限、APIエラー、429表示、検証済みリンク遷移を確認する。

### Lambda

- 入力検証、過大な履歴の拒否、現在ルート照合を単体テストする。
- 日本語キーワード、FAQ、現在ページを使った関連項目選択をテストする。
- 最低スコア未満ではOpenAIを呼ばず固定Contact回答を返し、採用した上位5件以外がモデル入力へ入らないことを確認する。
- 未知のページIDと今回モデルへ送っていないページIDを破棄し、許可集合だけをURLへ変換する。
- 日次上限とセッション上限を1つのトランザクションで確保し、並行リクエストでも超えないこと、上流失敗時も枠を返却しないことを確認する。
- OpenAI成功、拒否、タイムアウト、不正な構造化出力をモックして確認する。
- システム指示の上書き、任意URLの生成、未登録ページへの誘導を求める入力でも、検証済みリンク以外を返さないことを確認する。
- LambdaとCDKには既存のテスト基盤がないため、選択器、入力検証、リンク変換、利用上限、OpenAIモックを実行できる最小のTypeScriptテスト構成を追加する。
- テストでは実際のOpenAI APIを呼ばず、最終スモークテストだけを明示的に1回実行する。

### 全体

- フロントエンドの関連Vitest、lint、production buildを実行する。
- CDKのTypeScript buildとsynthを実行する。
- synthされたテンプレートで `/assistant/POST` の2/4スロットリング、Assistant専用CORS、シークレットとDynamoDBの最小権限を確認する。
- APIキーがビルド成果物とログに含まれないことを確認する。
